## 25. The Professional's Mindset

**Problem:** Developers think they're done when code runs.
**Mainstream:** Working code = mission accomplished.
**Their approach:** Professional not done until: tested, refactored, documented, reviewed. Code is beginning, not end.
**Example:** Code review finds issues. Refactor based on feedback. Now professional quality.

## 26. Lazy Evaluation as Functional Virtue

**Problem:** Intermediate computations waste memory. Eager evaluation forces all work upfront.
**Mainstream:** Compute everything, store intermediate results in collections.
**Their approach:** Streams in Java are lazy by design — nothing executes until a terminal operation triggers it. Compose transformations without materializing intermediate data structures.
**Example:** `.filter().map().limit(5)` on a million-element stream evaluates only the items needed to produce five results. Laziness enables efficient composition.

## 27. Shared Mutable State Is the Enemy of Concurrency

**Problem:** Concurrent programs fail mysteriously with data corruption and race conditions.
**Mainstream:** Add locks and synchronization to shared objects.
**Their approach:** Synchronization is a workaround, not a solution. The real enemy is shared mutable state. Either make state immutable or don't share it. Eliminate the problem, don't manage it.
**Example:** A mutable list shared across threads — protected by locks — is always one missed lock away from a race condition. An immutable list shared across threads needs no locks at all.

## 28. Lambdas Are Values, Not Shorthand

**Problem:** Developers treat lambdas as shorter anonymous inner classes and miss their real power.
**Mainstream:** Use lambdas to reduce boilerplate syntax.
**Their approach:** A lambda is a value representing a computation. Store it, pass it, return it, defer it. This changes how you design APIs — computation itself becomes a parameter.
**Example:** `() -> expensiveComputation()` stores a deferred action as a value. A lambda passed to a library means "execute this later under these conditions" — not just shorter syntax but a new design model.

## 29. Pure Functions Enable Trivial Testing

**Problem:** Testing requires mocking dependencies, complex setup, and brittle teardown.
**Mainstream:** Mock all dependencies to isolate the unit under test.
**Their approach:** Pure functions — same input always produces same output, no side effects — are trivially testable. No mocks, no setup, no teardown. Design towards purity at boundaries.
**Example:** `int add(int a, int b)` needs one test: call it, check the result. `void addToDatabase(int a)` needs a database mock. Design for pure functions whenever possible.

## 30. Optional Is for Return Types Only

**Problem:** Null causes NullPointerExceptions. Developers sprinkle Optional everywhere as a fix.
**Mainstream:** Use Optional wherever a value might be absent.
**Their approach:** Optional has one purpose: communicating to callers that a return value might be absent. Never use Optional as a method parameter or field type. That misuse defeats its purpose and adds noise.
**Example:** `Optional<User> findById(int id)` — correct. It tells callers to handle the absence case. `void save(Optional<User> user)` — wrong. The caller should never create an Optional to pass in; just check before calling.

## 31. The Hollywood Principle: Inversion of Control

**Problem:** Application code directly calls framework code. Testing and reuse are hard.
**Mainstream:** Your code orchestrates everything, calling libraries and frameworks when needed.
**Their approach:** "Don't call us, we'll call you." Inversion of Control means the framework calls your code. You provide callbacks and implementations; the framework orchestrates execution.
**Example:** A servlet container calls your `doGet()` — you don't call the container. Spring calls your `@Bean` methods. You define what to do; the framework decides when to do it.

## 32. Tell, Don't Ask

**Problem:** Objects expose internal state. External code makes decisions based on that state, creating fragile coupling.
**Mainstream:** Use getters to retrieve state, then make decisions outside the object.
**Their approach:** Tell objects what to do; don't ask for their data to decide for them. Behavior belongs where the data lives. Objects that expose state for external logic violate encapsulation.
**Example:** Instead of `if (account.getBalance() > amount) { account.setBalance(account.getBalance() - amount); }`, just tell the account: `account.debit(amount)`. The account handles its own invariants.

## 33. Excessive Mocking Is a Design Smell

**Problem:** Tests become long setup rituals with many mocked dependencies.
**Mainstream:** Mock all external dependencies when writing unit tests.
**Their approach:** Excessive mocking is not a testing problem — it is a design problem. A class needing five mocks has five dependencies and is doing too much. Redesign before mocking.
**Example:** A service class that needs mocks for a database, a cache, an email sender, a logger, and an audit trail has five reasons to change. Split responsibilities; each piece becomes testable with little or no mocking.

## 34. Comments Explain Why, Not What

**Problem:** Code is cluttered with comments restating what the code already shows.
**Mainstream:** Comment code thoroughly so others can understand it.
**Their approach:** If you need a comment to explain what code does, rename the code or extract a method. Comments explaining the "why" — business rules, legal requirements, non-obvious decisions — are valuable. Comments explaining the "what" are a failure of naming.
**Example:** `// iterate over users` before a for loop is redundant. Extract to a method `processActiveUsers()` instead. But `// required by regulation XYZ — cannot be removed` explains context that code cannot express.

## 35. The Open/Closed Principle With Functions

**Problem:** Adding new behavior requires modifying existing classes or creating sprawling inheritance trees.
**Mainstream:** Use abstract classes and subclasses to extend behavior.
**Their approach:** Pass functions as parameters to keep existing code closed for modification while opening it for extension. A lambda argument lets callers supply new behavior without touching the host code.
**Example:** A `sort(list, comparator)` method is open for any sorting strategy by accepting a comparator function. Existing sort code is never modified, yet behavior is completely customizable.

## 36. Data Classes Remove Noise From Signal

**Problem:** Java domain classes are 80% boilerplate — constructors, getters, equals, hashCode, toString — hiding what actually matters.
**Mainstream:** Generate the boilerplate with IDEs or Lombok.
**Their approach:** Boilerplate is noise that hides signal. Kotlin's `data class Point(val x: Int, val y: Int)` expresses everything meaningful in one line. Less noise means the important code is actually readable.
**Example:** A Java `Person` class with three fields takes 40 lines. The Kotlin data class takes one line. Everything generated — equals, hashCode, copy, toString — and the intent is instantly clear.

## 37. Extension Functions Over Utility Classes

**Problem:** Static utility classes are a design antipattern — `StringUtils.pad(str)` — necessary only when you can't add methods to types you don't own.
**Mainstream:** Create utility classes with static methods for helper functionality.
**Their approach:** Extension functions (Kotlin, Groovy, Scala) let you add methods to existing types without subclassing. `"hello".isPalindrome()` reads naturally and keeps the behavior where discovery expects it.
**Example:** Kotlin extension: `fun String.isPalindrome() = this == this.reversed()`. Now every String has this method. No utility class, no static import, no awkward call site.

## 38. Senior Engineers Think Longer Time Horizons

**Problem:** Code that works today creates problems that accumulate invisibly until they become crises.
**Mainstream:** Build what works now; deal with future problems when they arrive.
**Their approach:** Senior engineers habitually ask questions that junior engineers skip: What happens when this fails? What does this look like in six months? How will someone debug this at 3am? This longer horizon shapes fundamentally different design choices.
**Example:** Junior engineer: "The sort works." Senior engineer: "The sort works — but what happens with 10 million records? What if the input is already sorted? What is the expected shape of bad data?"

## 39. Deliberate Practice vs. Accumulated Experience

**Problem:** Years of writing similar code produces experience without mastery.
**Mainstream:** The more code you write, the better you get.
**Their approach:** Experience without deliberate practice leads to plateau. Deliberate practice — focused repetition with conscious attention to improvement and feedback — produces mastery. Coding katas done intentionally are more valuable than years of routine work.
**Example:** Doing the same string-manipulation kata ten different ways with different constraints — immutability only, no loops, minimum lines — teaches technique that daily work never forces you to develop.

## 40. Backpressure in Reactive Systems

**Problem:** Fast producers overwhelm slow consumers. Buffers explode, systems crash under load.
**Mainstream:** Buffer all events; consumers catch up when they can.
**Their approach:** Backpressure lets consumers signal their capacity back to producers. Reactive streams built on backpressure don't collapse — the producer rate is governed by what downstream can absorb.
**Example:** A reactive subscriber requests ten items at a time. The producer sends ten, waits for the subscriber to request more. No unbounded buffering. The system stays stable regardless of producer speed.

## 41. Functional Composition Over Inheritance Hierarchies

**Problem:** Class inheritance creates rigid, brittle hierarchies that are hard to extend without modification.
**Mainstream:** Extend behavior by subclassing existing classes.
**Their approach:** Small, single-purpose functions composed together are more flexible than inheritance hierarchies. Each function is independent, testable, and combinable in ways a class tree cannot be.
**Example:** `list.filter(activeUsers).map(toEmailAddress).sorted()` composes three independent operations. Doing this with inheritance — `SortedEmailMappedActiveUserList extends ...` — is unmaintainable.

## 42. Lambdas Collapse Design Patterns

**Problem:** Gang of Four patterns require separate interface definitions, abstract classes, and boilerplate implementations.
**Mainstream:** Implement Strategy, Command, Observer as formal class hierarchies.
**Their approach:** Many GoF patterns are simply "pass a function." Lambdas collapse the Strategy pattern to a method parameter. The design pattern is still present conceptually, but the ceremony disappears.
**Example:** Strategy pattern pre-Java 8: define an interface, create an anonymous class, instantiate and pass it. Java 8+: pass a lambda. Same pattern, one-tenth the code, same intent conveyed.

## 43. Types as Living Documentation

**Problem:** Written documentation falls out of sync with code. Comments lie.
**Mainstream:** Document methods with Javadoc and inline comments.
**Their approach:** Express intent through types. A method signature is documentation the compiler enforces. `Optional<User>` says more than `@return may be null`. A well-typed system is self-documenting and accurate by definition.
**Example:** `Duration` instead of `int milliseconds` makes units self-evident. `NonEmptyList<T>` communicates the list will never be empty. The type constrains AND documents simultaneously.

## 44. Immutability as Default Posture

**Problem:** Mutable state creates unexpected, hard-to-trace bugs — especially across time and threads.
**Mainstream:** Make things mutable by default; add `final` when you're sure you need it.
**Their approach:** Default to immutability. In Kotlin, prefer `val` over `var`. Every `var` is a question: why does this need to change? Mutability should be explicit, deliberate, and rare.
**Example:** A Kotlin data class with `val` fields cannot be mutated after construction. When you need to "change" a value, you create a new instance with `.copy(field = newValue)`. Bugs from unexpected mutation become impossible.

## 45. Right Level of Abstraction

**Problem:** Code is either too specific (copy-paste repetition) or too abstract (generic frameworks built for one use case).
**Mainstream:** Abstract at the first sign of repetition, or build for all imaginable futures.
**Their approach:** Abstract at the level the problem requires — no higher, no lower. Too-early abstraction creates complexity you cannot predict. Wait until you see the pattern repeat clearly before naming and extracting it.
**Example:** Writing a generic query DSL before you have three actual queries produces a framework you'll spend more time maintaining than using. Write the concrete queries first; the right abstraction reveals itself.

## 46. Error Handling at the Boundary

**Problem:** Checked exceptions break functional composition. Stream lambdas that throw checked exceptions require ugly try-catch wrappers.
**Mainstream:** Wrap checked exceptions inside lambdas and rethrow as unchecked.
**Their approach:** Handle exceptions at the system boundary, not inside functional pipelines. Inside a composition, errors should flow as values — Optional, Result types, or monadic wrappers — not as control-flow exceptions.
**Example:** A stream of file reads that might fail: wrap the file-reading logic in a helper that returns `Optional<String>`, catching the IOException internally. The stream composition stays clean; failure is represented as data.

## 47. API Ergonomics and Discoverability

**Problem:** Powerful APIs are discovered through documentation reading, not exploration. Wrong usage compiles fine and fails at runtime.
**Mainstream:** Build a flexible API; write documentation explaining correct use.
**Their approach:** Good API design makes the simple case simple, makes wrong usage hard or impossible, and reveals the next step through structure and naming. Discoverability through IDE completion is a design goal.
**Example:** A builder pattern — `User.builder().name("Alice").email("a@b.com").build()` — guides usage through method chaining. Each method reveals what's available next. Missing a required field fails at build time.

## 48. Cognitive Load Is a Real Cost

**Problem:** Complex code is expensive to maintain because it is expensive to understand.
**Mainstream:** Complexity is unavoidable in complex systems.
**Their approach:** Cognitive load — the mental effort needed to understand code — is a concrete, measurable cost. Every extra variable, branch, and dependency increases it. Reduce cognitive load deliberately, just as you would reduce coupling.
**Example:** A function with three parameters, two branches, and one return path fits in working memory. A function with eight parameters, five branches, and three early returns does not. The latter will be misunderstood, and misunderstanding produces bugs.

## 49. Design for Testability From the Start

**Problem:** Retrofitting tests onto existing, untestable code is expensive and often incomplete.
**Mainstream:** Write code, then write tests for it.
**Their approach:** Code that is easy to test is usually better designed — more focused, less coupled, with clearer dependencies. Thinking about testability during design produces better design, not just better tests.
**Example:** A method that accepts an interface instead of a concrete class is immediately testable — inject a test implementation. A method that creates its own dependencies is untestable without modifying the method. Testability and good design are the same thing.

## 50. AI Tools Are a Pair Programmer, Not an Oracle

**Problem:** Developers accept AI-generated code without understanding it, producing code they cannot maintain or defend.
**Mainstream:** Use AI to generate code quickly; review lightly or not at all.
**Their approach:** AI tools are like a fast-typing pair programmer — good at syntax, good at common patterns, but wrong about context and edge cases in ways you must catch. You remain responsible for understanding the code: why it is correct, what it misses, how it fits the design.
**Example:** AI generates a sorting function. You must still ask: Is this the right algorithm for the data size? Does it handle empty input? Does it match the contract callers expect? Speed of generation does not remove the obligation to understand.
