# Saptarshi Prakash (Indian) — Full Learnings

## 1. UX Choreography: Motion Elevates Understanding

**Problem:** UI static. Users don't understand transitions and interactions.
**Mainstream:** Motion is decoration. Focus on static UI.
**Their approach:** Motion is fundamental interaction language. Choreography of micro-interactions guides understanding.
**Example:** Animation showing how menu appears teaches user interaction possibility better than static mockup.

## 2. User Natural Frequency Sync

**Problem:** Features designed without understanding how users think.
**Mainstream:** Design for average user. Assume standard cognition.
**Their approach:** Every user has natural frequency—how they naturally process, decide, act. Design syncs with their rhythm.
**Example:** Feedback timing matching user's expectation speed. Too fast feels untrustworthy; too slow feels broken.

## 3. Preemptive UX: Features Before Need

**Problem:** Users frustrated by friction until feature is added.
**Mainstream:** Add features when users request them.
**Their approach:** Anticipate need. Build capability before user consciously knows they need it.
**Example:** Auto-save feature exists before user realizes they need it. Removes frustration point.

## 4. AI-Native Design: Designing Agentic Workflows

**Problem:** AI treated as autocomplete. Not as agent.
**Mainstream:** AI improves existing workflows. Humans still primary agent.
**Their approach:** Design where AI is agent: it initiates, proposes, learns from feedback. Humans supervise and direct.
**Example:** System that proposes next action based on pattern. User approves or redirects. AI learns.

## 5. Intelligence Layers Over UI Layers

**Problem:** Intelligence buried in features. Users don't understand system.
**Mainstream:** Better features = better product.
**Their approach:** Intelligence should be visible in interface. System shows its reasoning. User understands why it suggests something.
**Example:** Recommendation with explanation: 'Based on your reading patterns, you might like this.' Transparency builds trust.

## 6. Designing for Discovery and Reasoning

**Problem:** Product supports task completion, not learning or discovery.
**Mainstream:** User comes with intent. Complete intent.
**Their approach:** Product should enable discovery: learning new capabilities, new ways to use features. Support reasoning.
**Example:** Tips system that teaches advanced techniques. Help users discover depth beyond basic usage.

## 7. Micro-Experiences as Impact Points

**Problem:** Only major interactions matter. Miss small moments.
**Mainstream:** Focus on major user journeys.
**Their approach:** Micro-interactions (hover state, loading animation, confirmation message) carry disproportionate impact.
**Example:** Satisfying delete animation with undo possibility. Small moment changes user feeling about product.

## 8. Effort-Reward Ratio Timing

**Problem:** Users invest effort without feeling rewarded.
**Mainstream:** Minimize effort. Show reward eventually.
**Their approach:** Time rewards close to effort. Visual/emotional reward should follow action quickly.
**Example:** Upload completes and immediately you see preview. Effort rewarded instantly, not minutes later.

## 9. Problem-Solving as Core Experience

**Problem:** Product optimized for efficiency, not for user engagement.
**Mainstream:** Frictionless experience. Minimal steps.
**Their approach:** Optimal engagement comes from solving problems. Product should engage mind, not just streamline tasks.
**Example:** Analytics dashboard that shows data and invites interpretation. User solves problems with data.

## 10. Motion Hierarchy for Information Design

**Problem:** Moving elements create noise. Distracting.
**Mainstream:** Minimize motion. Keep focus on content.
**Their approach:** Motion hierarchy: what moves first, what follows, what stays still. Motion shows importance and flow.
**Example:** In dashboard: important metrics animate, secondary metrics static. Motion directs attention.

## 11. Designing for Indian Users at Scale

**Problem:** Products designed for global average. Fail locally.
**Mainstream:** Global design. Local tweaks.
**Their approach:** Understand Indian user context: literacy levels, connection speeds, data costs, usage patterns. Design from there.
**Example:** Low-data mode isn't feature; it's core design constraint. Product works on 2G.

## 12. Feedback Loops as Learning Tools

**Problem:** System acts on user input. User doesn't learn from outcomes.
**Mainstream:** Process user input. Execute.
**Their approach:** Design feedback loops that teach user: show how action changed system, what it means.
**Example:** When user adjusts setting, show immediate effect. Feedback loop teaches cause-effect.

## 13. Constraint as Feature

**Problem:** Users want unlimited options. Paralyzed by choices.
**Mainstream:** Enable everything. Empower user.
**Their approach:** Strategic constraint simplifies choice. Defaults, limited options, smart suggestions reduce cognitive load.
**Example:** Color picker with curated palette instead of infinite colors. User makes better choice faster.

## 14. Pattern Recognition in Interface Design

**Problem:** Users learn through explanation. Intuition incomplete.
**Mainstream:** Make interface intuitive. Expect quick understanding.
**Their approach:** Users learn patterns through use. Design patterns consistently so users build mental models.
**Example:** Button style consistent everywhere. Users learn: 'this style = tappable.' Pattern recognition enables fluency.

## 15. Error States as Conversation

**Problem:** Errors are interruptions. User frustrated.
**Mainstream:** Prevent errors. Show error, move on.
**Their approach:** Error state as conversation: what went wrong, why, what now? Teach user through error.
**Example:** Form error message that explains 'Email must include @ symbol' teaches rule, not just rejects input.

## 16. Onboarding as Experience Design

**Problem:** Onboarding is tutorial. Boring.
**Mainstream:** Explain features. Move to product.
**Their approach:** Onboarding is experience design. Delight users with product's personality while teaching.
**Example:** Onboarding that gamifies learning: complete tutorial activities, earn points, unlock features.

## 17. Data Visualization as Storytelling

**Problem:** Charts show data. Don't tell story.
**Mainstream:** Accurate visualization. Data integrity.
**Their approach:** Data visualization should tell story: what's the insight? What should user understand? Design to reveal.
**Example:** Chart that highlights trend, adds context, guides interpretation. Data becomes story.

## 18. Accessibility as Design Foundation

**Problem:** Accessibility retrofitted. Excludes people.
**Mainstream:** Design for majority. Add accessibility later.
**Their approach:** Accessibility is core design principle. Color contrast, readable type, logical structure benefit everyone.
**Example:** High-contrast mode isn't special feature; it's how product is designed. Everyone benefits.

## 19. Personalization Versus Privacy

**Problem:** Personalization creepy. Privacy versus usefulness.
**Mainstream:** Personalize everything. Show capability.
**Their approach:** Transparent personalization: show user what data is used, why, enable opt-out. Trust through transparency.
**Example:** System says 'Based on your history, I recommend...' User understands, appreciates or disables.

## 20. Testing as Design Process

**Problem:** Design finalized, then tested. Changes are fixes.
**Mainstream:** Design → Test → Fix.
**Their approach:** Testing throughout design. Early testing reveals misassumptions. Design evolves based on how users interact.
**Example:** User testing at sketch stage reveals misunderstanding. Design pivots early, before expensive execution.

## 21. Mental Model Alignment

**Problem:** Product logic doesn't match user's mental model. Confusing.
**Mainstream:** Teach user how product works.
**Their approach:** Align product logic with how users think. User mental model should match system model.
**Example:** File system hierarchy matches user's folder thinking. Alignment makes system intuitive.

## 22. Performance as Feature

**Problem:** Speed treated as technical metric. Impact underestimated.
**Mainstream:** Optimize behind scenes. User doesn't care how fast, just that it works.
**Their approach:** Speed is experiential feature. Fast load = trust, professional, competent. Slow load = broken, amateur.
**Example:** Page that loads in 1 second feels professional. Same content loading in 5 seconds feels broken.

## 23. Inclusive Design Thinking

**Problem:** Design for mainstream. Exclude edge cases.
**Mainstream:** 80% of users cover design. Edge cases handled later.
**Their approach:** Design for diversity: different abilities, contexts, literacies. Constraints of edge cases improve design for everyone.
**Example:** Designing for blind users (screen reader) improves semantic markup, benefits everyone.

## 24. Delight Through Unexpected Moments

**Problem:** Product functional but joyless.
**Mainstream:** Efficient interface. Speed to completion.
**Their approach:** Build moments of delight: unexpected haptics, animations, surprises. These create emotional connection.
**Example:** Shake animation with error feedback. Haptic feedback on success. Small moments create joy.

## 25. Designing for Behavioral Change

**Problem:** Product enables behavior change. But doesn't encourage it.
**Mainstream:** Provide functionality. User drives behavior.
**Their approach:** Design can encourage healthy behaviors: streaks, progress visualization, community, social proof.
**Example:** Habit app with streak counter, friend comparison, celebration of milestones. Design encourages consistency.

## 26. Always Design with Real Content

**Problem:** Designers use placeholder content (Lorem Ipsum, generic product images) that misrepresents how real content will behave.
**Mainstream:** Design with generic placeholder content, substitute real content at the end.
**Their approach:** Design decisions — information hierarchy, layout, trust signals — can change completely when real content is substituted. Test every design decision against the actual content that will appear in production.
**Example:** A listing for "Chicken Whopper from Burger King" looked elegant in a standard card layout. Substituting the real content — "Deluxe North Indian Egg Meal by Ms. Nidhi Parulkar" — revealed that the same design no longer worked: the name was too long, the context unfamiliar, the elegance gone. The design required rethinking. "Always design with real content. Decisions can massively change based on this." (Swiggy Design Medium, "How I designed the listing for a brand new Food App," 2019)

## 27. Food-First (Product-First) Information Hierarchy

**Problem:** Listing designs default to leading with the vendor or brand name, even when the product itself is the primary decision driver.
**Mainstream:** Organize listings by vendor or restaurant name, with product as secondary.
**Their approach:** When users are choosing based on the product rather than the vendor — particularly in new or unfamiliar categories — lead with the product. Put the food (or product) front and center: image, description, and key details first. Vendor information supports the product, not the reverse.
**Example:** Swiggy Daily's "food-first approach" organized listings around individual meal items rather than kitchen names, because users were selecting by what they wanted to eat, not who made it. This required a fundamentally different hierarchy than the restaurant-first structure used in the main Swiggy app. (Swiggy Design Medium, 2019)

## 28. Halo and Cheerleader Effects Shape Perceived Trust

**Problem:** Unknown or new vendors receive no trust from users, regardless of product quality, because there is no social proof or brand recognition.
**Mainstream:** Trust is built slowly over time through ratings and reviews.
**Their approach:** When unknown items are surrounded by familiar, trusted ones, trust transfers — the Halo Effect and Cheerleader Effect in action. Users become more confident about an unfamiliar option when it appears alongside known ones. But when all items are unknown (no halos available), a different trust-building strategy is required.
**Example:** On Swiggy's main app, a new unknown restaurant listed near well-known chains gets a trust boost from proximity. On Swiggy Daily, where all vendors were new home-chefs with no brand recognition, no halo existed — so each chef needed their own trust signal built from scratch. (Swiggy Design Medium, 2019)

## 29. Humanizing Unknown Vendors Builds User Trust

**Problem:** Users won't order from vendors they don't recognize and have no way to evaluate.
**Mainstream:** Use ratings, reviews, and certifications to establish credibility.
**Their approach:** When objective trust signals (ratings, fame, established brand) don't exist, humanize the vendor. Showing a chef's photo, a short personal story, and a unique credential makes the decision feel personal rather than anonymous. Users relate to a real person more readily than to an unverified entity.
**Example:** Adding a chef's photograph and a line like "YouTube cooking instructor" or "30-year-old family recipes" to each listing on Swiggy Daily gave users a human hook — something to connect to when choosing between unknown vendors offering the same dish. "How a simple humanising element can instil so much of empathy." (Swiggy Design Medium, 2019)

## 30. Iterative Design: Start With Proven Patterns, Then Challenge Them

**Problem:** Designers start from scratch or from abstract ideals, rather than grounding iteration in known-good baselines.
**Mainstream:** Ideate broadly, prototype multiple concepts, select best.
**Their approach:** Start with the most proven, tested pattern available — then introduce real content and real constraints to reveal where it breaks. Each iteration is driven by a specific failure discovered in the previous step, not by theoretical improvement. "I like to start with something tried-and-tested, and then build over it based on the need."
**Example:** The Swiggy Daily listing started with an exact copy of the Swiggy POP listing (tested, proven). Substituting real content in "Take 2" revealed the design broke under actual product names and descriptions. "Take 3," "Take 4," and the Final Take each solved one specific failure exposed by the previous version. (Swiggy Design Medium, 2019)

## 31. Physics Is the Foundation of Believable Motion

**Problem:** Motion designers treat animation as a stylistic layer without grounding it in how objects actually behave.
**Mainstream:** Learn animation software tools and apply transitions aesthetically.
**Their approach:** Real-world physics — how objects fall, move, absorb impact, and resist air — is the foundation for motion design that feels believable. Understanding these physical behaviors (shock absorbers, fluttering cloth, wheel rotation) is all the physics required to start. "I am not completely wrong when I say you start with Physics."
**Example:** Animating a delivery bike involves: wheels rotating at speed (showing blur not spokes), chassis heaving from shock absorbers, rider's body exhibiting humanly movements (elbows, neck, back), cap fluttering from wind resistance. Each animation is grounded in real physical cause-and-effect. (Swiggy Design Medium, "How I started Motion Design using high-school Physics," 2019)

## 32. Break Complex Animation Into Individual Components

**Problem:** Complex animations are attempted as a single unit, creating unmanageable choreography.
**Mainstream:** Animate the whole scene together; adjust timing globally.
**Their approach:** "Break down the problem into pieces — this time, quite literally." Each visual component is separated into its own layer and animated independently, then assembled. This enables precise control over individual elements, easier debugging, and reusable parts.
**Example:** The Swiggy delivery bike animation was broken into: chassis, front wheel, back wheel, rider torso, rider arms, rider neck/head, food box, and Santa cap. Each component was animated independently in After Effects — wheels rotating, chassis heaving, arms bending — then combined into the final scene. (Swiggy Design Medium, 2019)

## 33. Subtle Human Movements Add Disproportionate Life

**Problem:** Illustrations and character animations feel stiff because only obvious movements are animated.
**Mainstream:** Animate the large, visible actions (walking, waving) and leave subtle body mechanics static.
**Their approach:** Very subtle movements — the slight bend of elbows, the rotation of a neck, the stretch and flex of a back — add disproportionate life to character animations. These bits are often invisible in isolation but their absence is felt. They "clearly differentiate the lifeless food-box from the human, full of life."
**Example:** In the delivery bike animation, animating the rider's elbows shifting, neck tilting, and back flexing — movements almost too small to consciously register — transformed the rider from a static graphic into a living person. Without them, the animation felt plastic. (Swiggy Design Medium, 2019)

## 34. Controlled Exaggeration Is Allowed in Motion Design

**Problem:** Motion designers either exaggerate too much (cartoonish) or stay too rigid (lifeless).
**Mainstream:** Match real-world physics exactly; exaggeration is a sign of immaturity.
**Their approach:** Controlled exaggeration — exceeding realism slightly to make a point or improve visual clarity — is not only acceptable but often desirable. The key word is controlled. Exaggeration without restraint becomes noise; restraint without exaggeration produces static.
**Example:** The delivery executive's Santa cap was animated with more flutter than it would realistically exhibit at bike speed — and then pulled back slightly. "This step of fluttering the cap is highly addictive and you may feel very tempted to overdo it. In fact, I too overdid it! But hey… controlled exaggeration is allowed." (Swiggy Design Medium, 2019)

## 35. Relative Motion Indicates Forward Movement More Efficiently

**Problem:** Animating a character moving forward across a screen requires moving the entire scene, which is complex and often impractical.
**Mainstream:** Move the character or object forward in the frame.
**Their approach:** Move the background, not the character, to indicate forward movement. This technique from classic video games (Road Rash, Mario, Contra) is simpler to implement, loops cleanly, and reads as convincingly as true forward movement — because relative motion is how we perceive movement in the real world.
**Example:** The Swiggy delivery bike was held in the center of the frame while a looping cityscape moved behind it. "Relative motion, you see." The bike appears to travel forward without the animation requiring the character to move. The looping cityscape was created from two identical frames stacked end-to-end. (Swiggy Design Medium, 2019)

## 36. Design Doesn't Live in Figma — It Lives in Conversations

**Problem:** Design decisions are made and communicated primarily through static screens and design files.
**Mainstream:** Design = Figma output. Stakeholders evaluate polished mockups.
**Their approach:** Real design decisions are made in conversations — with engineers about feasibility, with PMs about business goals, with operations about real-world constraints, with customer support about where users actually struggle. The Figma file is documentation of what was decided in those conversations, not where the deciding happens.
**Example:** During the Swiggy Instamart rollout, the design team bridged the gap between product managers' business goals, engineers' feasibility concerns, and customer support's real pain points — not through presentations of screens but through structured conversations. "Design doesn't live in Figma; it lives in conversations." (Swiggy blog, "Building Beyond the Brief," 2025)

## 37. Innovation Through Nuance, Not Disruption

**Problem:** Design teams frame innovation as requiring reinvention of core flows and mechanics.
**Mainstream:** Innovation = changing how the fundamental product works.
**Their approach:** Bold, differentiated design rarely requires changing what the product does. The innovation lives in the details: a subtle twist in a familiar flow, a distinct micro-interaction, an unconventional copy tone that makes something feel fresh without confusing users who already know the patterns.
**Example:** Swiggy's Snacc app followed all established norms of food discovery and ordering mechanics. The differentiation was entirely visual — pushing boundaries in the visual layer without altering core functionality. "It's a great case of innovation through nuance, not disruption." (Swiggy blog, "Building Beyond the Brief," 2025)

## 38. Humans Are Inherently Good Problem-Solvers — Trust That Instinct

**Problem:** Design teams over-rely on frameworks, best practices, and formal processes, stifling the raw problem-solving instinct.
**Mainstream:** Follow a rigorous design process framework to ensure quality output.
**Their approach:** All humans — not just designers — are inherently effective problem-solvers. The instinct that makes someone reroute in the rain when their cab doesn't show is the same instinct that makes great design. At work, people create "invisible rules for themselves, over-indexing on frameworks and best practices that, while useful, can stifle raw creativity." The scrappy instinct should be brought to the early stages of every project.
**Example:** "Think about the everyday challenges we face. You miss your cab, it starts raining, and there's no umbrella. Five guests show up unannounced, and you've only got three laddoos. Somehow, we always find a way. These are everyday design problems, just without the Figma file." (Swiggy blog, "Building Beyond the Brief," 2025)

## 39. User-Friendly Is Not in Conflict with Business Returns

**Problem:** Design teams are asked to compromise user experience in favor of business or conversion metrics.
**Mainstream:** User needs and business goals are in tension; trade-offs must be made.
**Their approach:** Anything that is not user-friendly cannot generate business returns in the long run. When user needs conflict with business goals, starting with the user is not idealism — it is strategy. Products that consistently do what is right for the user create outcomes that naturally align with business, product, and team goals.
**Example:** Facing a difficult product decision where user needs clashed with business goals, Swiggy's CEO's guidance was: "Sapta, anything that is not user-friendly can never give us business returns in the long run." This became the lens Saptarshi applies to almost every design decision. "Start with the user. Always." (Swiggy blog, "Building Beyond the Brief," 2025)

## 40. AI as Anticipatory Design: The Invisible Interface

**Problem:** AI features are added as visible tools — chatbots, suggestion widgets — rather than woven into the experience.
**Mainstream:** AI as a visible layer users interact with explicitly.
**Their approach:** The most powerful AI integration makes the interface feel invisible — an ultra-efficient assistant that anticipates needs without being asked. By learning context (past behavior, time of day, weather, social patterns), AI transforms a transactional product into one that feels intuitive. "Interfaces don't just respond, they anticipate."
**Example:** Swiggy's AI design vision: "By learning from past orders, the time of day, the weather outside, or even what your friends are eating, AI can transform a transactional service into something that feels intuitive and almost invisible, like an ultra-efficient butler — think Alfred from Batman — who knows your preferences inside out. You don't have to spell things out. A slight nod, a wave, or a word or two would be enough." (Swiggy blog, "Building Beyond the Brief," 2025)

## 41. Disagree Openly, Then Commit Fully

**Problem:** Teams either suppress disagreement (false consensus, weak decisions) or sustain disagreement past the decision point (fractured execution).
**Mainstream:** Seek consensus before moving forward; defer to whoever has authority.
**Their approach:** Create a culture where disagreement is welcomed and expected during deliberation — then, once a direction is chosen, everyone commits fully regardless of whose idea won. This combination of open dissent and unified execution produces both better decisions and stronger follow-through. "Stand up, disagree, and fully commit."
**Example:** Swiggy's design culture: "There's space to challenge ideas, ask questions, and offer a different lens. Once a direction is chosen, no matter whose idea it was, we move forward together with clarity, conviction, and a shared sense of ownership." This is described as the value Saptarshi resonates with most. (Swiggy blog, "Building Beyond the Brief," 2025)

## 42. Small Design Decisions Forge Habits That Outlast Their Products

**Problem:** Designers underestimate the cumulative, long-term impact of small UI choices on user behavior.
**Mainstream:** Focus design attention on major flows and key moments.
**Their approach:** A subtle change in design can have long-lasting impacts on user minds — creating new habits that outlast the feature, the app, or even the company. "The flap of the butterfly wing, if designed correctly, can cause a typhoon of changes beneficial to us." Designers shape behavior at scale and across time, not just sessions.
**Example:** From TEDxIIITBangalore: designing a subtle friction point or reinforcement in the right moment can forge a new habit in users — and those habits "could survive generations." The implication: small, thoughtful design decisions compound in ways that large, flashy ones often don't. (TEDx talk, "Designing the Future Through the Little Things," September 2020)

## 43. Question Rules Rather Than Follow Them — Including Design Rules

**Problem:** Design practitioners treat best practices as fixed rules, applying them without examining whether they apply to the specific context.
**Mainstream:** Follow established design conventions and frameworks.
**Their approach:** Rules are patterns that work for the majority of cases. They are not universal laws. Every rule must be questioned: Why does this rule exist here? What problem does it solve? Is that problem present in this specific context? Only by understanding the reason behind a rule can a designer know when to apply it, when to adapt it, and when to discard it.
**Example:** "Even if I know the rules I always question — why is this rule followed here? What is the reason that this step exists? I believe that rules are nothing but a set of common things, which works for majority of the people." — from 8px Magazine interview (2020). This questioning approach is how Saptarshi, as a self-taught designer, developed genuine mastery: not by memorizing conventions, but by understanding why they exist.
